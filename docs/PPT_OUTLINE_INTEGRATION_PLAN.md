# PPT 逐页大纲流程接入计划

## 背景与目标

当前流程在资料分析后仅通过整体计划确认进入内容与 SVG 生成，尚未提供独立的逐页大纲展示、修改与再次确认机制。用户若要调整某一页的内容计划，只能通过确认接口的 `comment` 或 `answers` 传递自然语言意见；同时，`approved=false` 会使任务失败，不能形成“修改后重新确认”的闭环。

本计划的目标是在生成 SVG 前建立可审阅、可修改、可锁定的逐页大纲，并使其成为后续设计规格、文生图、备注和 SVG 生成的共同依据。

## 目标流程

```text
资料导入与分析
  → 整体方向确认（可选）
  → 逐页大纲生成
  → 逐页修改与确认
  → 最终计划锁定（outline.json / design_spec.md / spec_lock.md）
  → 文生图（IMAGE_ENHANCED 模式）
  → 备注与 SVG 生成
  → 校验、组装与 PPTX 导出
```

## 分阶段实施

### 第一阶段：最小可用的大纲确认

目标：让用户在生成 SVG 前看到并调整每页计划。

1. Agent 在资料分析后生成逐页大纲，并通过 `request_plan_confirmation` 的 `contextData` 返回。
2. 每页大纲至少包括：`slideNo`、`title`、`keyMessage`、`bullets`、`visualSuggestion`。
3. 前端按页展示大纲，支持填写整体意见和指定页修改意见。
4. 用户确认后，Agent 将确认意见落实到 `design_spec.md` 与 `spec_lock.md`，再开始后续生成。
5. 用户要求修改时，任务应进入大纲修订，而不是直接失败；Agent 根据意见重生成大纲并再次请求确认。

验收标准：用户能指定第 N 页的修改要求，并能在 SVG 生成前确认该修改已体现在新大纲中。

### 第二阶段：显式大纲业务节点

目标：使任务状态、失败恢复和前端交互能够明确识别大纲阶段。

建议新增节点：

```text
PROJECT_READY
  → OUTLINE_DRAFTED
  → OUTLINE_CONFIRMED
  → DESIGN_SPEC_WRITTEN
  → SPEC_LOCK_WRITTEN
  → …
```

建议新增确认阶段标识 `outline_confirmation`。失败恢复时，可从已确认的大纲继续执行，不需要重新分析全部资料。

需要明确 `PLAN_CONFIRMED` 的职责：可将其保留为整体方向确认，也可逐步由 `OUTLINE_CONFIRMED` 取代，避免两个确认节点语义重叠。

### 第三阶段：结构化大纲编辑契约

目标：用稳定的数据契约替代仅依赖自然语言评论的修改方式。

定义 `PptOutline` / `SlideOutline` 数据模型，并在项目工作区持久化为 `outline.json`。示例：

```json
{
  "version": 1,
  "slides": [
    {
      "slideNo": 1,
      "title": "市场背景",
      "keyMessage": "目标市场处于增长窗口",
      "bullets": ["趋势", "机会", "结论"],
      "visualSuggestion": "增长曲线与关键数据",
      "imageRequirement": null
    }
  ]
}
```

前端以按页的新增、删除、修改操作提交变更；服务端将结构化编辑结果回传给 Agent。Agent 负责校验页码、顺序和必填字段，生成新的大纲版本后再次请求确认。

### 第四阶段：版本、差异与锁定

目标：支持多轮修改，并确保后续产物可追溯。

1. 为大纲引入递增的 `version`。
2. 每次修订后展示与上一版本的差异，包括新增页、删除页以及标题、结论、视觉方案的变化。
3. 用户确认后将对应版本标记为锁定。
4. `design_spec.md` 与 `spec_lock.md` 应记录所依据的已锁定大纲版本。
5. 大纲锁定后的再次修改应提示影响范围，包括相关图片、备注及 SVG 页面是否需要重新生成。

### 第五阶段：与文生图流程联动

目标：避免大纲尚未确定时提前生成图片，造成无效成本。

在 `IMAGE_ENHANCED` 模式中，将流程调整为：

```text
大纲确认 → 设计规格 → 图片清单确认/生成 → 图片确认 → SVG 生成
```

图片需求应来自已锁定大纲的 `imageRequirement`，而不是在 SVG 阶段临时决定。

## 推荐落地顺序

优先实施第一、二阶段：它们能以较小改动解决“修改某页输出计划”的问题，同时建立确认阶段和恢复节点的基础。待前后端确认交互稳定后，再实施结构化编辑、版本差异和文生图联动。

## 关键约束

- 未确认的大纲不得作为 SVG、备注或图片生成的最终依据。
- “要求修改”与“终止任务”必须是两种不同操作；只有用户明确终止时才将任务标记为失败或取消。
- 已锁定大纲是 `design_spec.md`、`spec_lock.md` 和后续输出的一致性来源。
- 保持 BASIC 与 IMAGE_ENHANCED 两种模式共用大纲确认契约；仅文生图相关字段和后续节点按模式启用。
